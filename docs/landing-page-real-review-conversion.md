## Problem Statement

商家买的不是一个 AI 生成工具，也不是一张 NFC 卡。商家买的是一个线下服务流程：服务员请愿意帮忙的真实顾客写评价，客户端页面帮顾客省掉两件麻烦事：

1. 不想动脑子组织评价内容。
2. 不想自己找 App 和店铺页。

因此落地页的任务不是说服顾客评价，而是在服务员已经发起请求之后，把顾客从“我愿意帮你评一下”送到“文案已复制，店铺页已打开”。

## Demand Evidence

- 用户明确选择商业化目标：帮商家持续拿到更多真实评价的服务。
- 用户明确平台选择由顾客自己完成，因为顾客知道“在哪里买的团购，就发到哪里”。
- 用户明确当前平台链接只能打开店铺页，评价入口由服务员在旁边协助寻找，页面不承担 App 内路径教学。
- 用户明确页面不写回报、优惠、领券或福利；这些属于服务员和顾客的现场交互，不进入客户端页面。

## Status Quo

当前客户端落地页已经具备正确的底层流程：

- `/landing/:token` 打开门店落地页。
- 初始接口只返回门店、关键词、图片、可用平台，不先发放评价。
- 顾客选择平台后，后端按 `platformCode` 从对应平台评价池发放文案。
- 顾客可选择关键词、换文案、编辑文案、复制，并点击平台链接。
- `platform_link_click` 被商家后台统计为“引导发布次数”。

当前问题不在后端主路径，而在客户端表达仍像工具页面：先“选择平台”，再“生成/换一换”，再“复制/去发布”。真实线下场景里，服务员已经在旁边等待，页面应该更像一个 10 秒完成的协助手势，而不是一个需要顾客理解的系统。

## Target User & Narrowest Wedge

目标用户不是所有浏览者，而是被服务员现场请求、愿意帮忙写真实评价的到店顾客。

最窄场景：

- 餐饮、美甲、足疗、美发等线下门店。
- 顾客来自美团团购、大众点评到店、小红书种草、抖音团购等来源。
- 服务员已经说过：“方便帮我们写个真实评价吗？贴一下这个卡片，系统会先帮你整理一版，你可以改成自己的话。”
- 顾客愿意帮忙，但不想自己想评价怎么写。

## Constraints

- 页面不出现回报、优惠券、小礼品、返券等内容。
- 页面不写 App 内评价路径，因为服务员现场协助。
- 页面必须持续强调“可以改成自己的话”，降低假评感。
- 平台选择保留，但表达成消费来源，不表达成系统配置。
- 主动作是 `复制并打开{平台}店铺`，不是单独的“复制”或抽象的“去发布”。
- 不能承诺真实发布成功；系统只能记录点击平台店铺页。
- 当前后端只能确认 `platform_link_click`，无法确认顾客是否在平台内完成提交。

## Premises

1. 顾客不是被页面说服才评价的，而是已经被服务员请求过、愿意帮忙。
2. 平台选择要保留，但用消费来源表达：美团团购来的、大众点评到店的、小红书看到来的、抖音团购来的。
3. 页面不提示回报，也不承诺任何奖励，避免把真实评价变成利益交换。
4. 店员会在旁边协助顾客找到平台内评价入口，所以页面不写 App 内路径教程。
5. 主动作是 `复制并打开{平台}店铺`，因为顾客最大的阻力是找 App 和动脑写。
6. 文案必须一直强调“可改成自己的话”，这是降低假评感的核心。

## Approaches Considered

### Approach A: 最小改版

保留现有结构，只改文案和按钮。平台按钮改成“美团团购来的 / 大众点评到店的”，`换一换` 改成 `换个说法`，主按钮改成 `复制并打开{平台}店铺`。

Effort: S
Risk: Low

Pros:
- 最快上线，基本不碰数据结构。
- 不推翻当前“先选平台，再发放对应文案”的正确路径。
- 适合先拿去真实门店观察 5-10 个顾客。

Cons:
- 页面仍偏功能步骤，服务员协同场景不够清晰。
- 复制和打开平台的完成反馈还不像一条服务流程。

Reuses:
- `frontend/src/views/landing/LandingPage.vue`
- `platformLinks`
- `POST /api/public/landing/:token/switch-review`
- `POST /api/public/landing/:token/events`

### Approach B: 理想转化版

把落地页重排成 10 秒完成任务：首屏接服务员话术，第一步选消费来源，第二步选体验重点，第三步展示已整理文案，底部只有一个主按钮 `复制并打开{平台}店铺`。

Effort: M
Risk: Med

Pros:
- 最贴合真实使用场景。
- 顾客不会感觉在操作 AI 工具，而是在完成一个很轻的帮忙动作。
- 商家后台的“引导发布次数”能更干净地对应主按钮点击。

Cons:
- 需要认真处理空状态：没平台、没文案、复制失败、跳转链接缺失。
- 要调整页面层级和交互，不只是改几个字。

Reuses:
- 现有平台选择和平台隔离评价池。
- 现有关键词标签。
- 现有复制工具和平台跳转工具。
- 现有 `platform_link_click` 统计口径。

### Approach C: 服务员协同版

页面显式做成“服务员陪同流程”：文案更短，按钮更大，完成后出现“已复制，交给店员协助发布”的状态。页面不教 App 路径，但承认服务员在场。

Effort: M
Risk: Med-High

Pros:
- 最符合服务员在旁边等待的线下真实流程。
- 顾客完成复制后不会迷失，知道下一步是服务员协助。
- 对培训店员有帮助，页面和话术一致。

Cons:
- 如果有些商家没有店员陪同，页面会显得不自助。
- “交给店员协助”写重了，会让顾客觉得被盯着。

Reuses:
- Approach B 的全部能力。
- 现有事件上报和商家价值看板。

## Recommended Approach

选择 Approach B: 理想转化版。

理由：A 太轻，只能修表面；C 太依赖每家店员执行。B 保留顾客自选平台，但把页面从“工具操作”改成“省脑子写真实评价”。这是当前商业定位下最稳的客户端形态。

## UX Contract

### 店员话术

店员现场只说两句：

```text
方便帮我们写个真实评价吗？
贴一下这个卡片，系统会先帮你整理一版，你可以改成自己的话。
```

页面首屏必须接住这句话，不重新解释 NFC，也不解释服务流程。

### 首屏文案

建议：

```text
帮你整理一版真实评价

选一下你是从哪里来的，我们先帮你写好。
你可以直接用，也可以改成自己的话。
```

不建议：

- “AI 帮你生成评价”
- “几秒生成好评”
- “领取奖励”
- “发布后找店员核销”
- “NFC 碰一碰体验”

### 页面流程

```text
1. 你是从哪里来的？
   美团团购来的 / 大众点评到店的 / 小红书看到来的 / 抖音团购来的

2. 这次体验想提哪点？
   团购划算 / 服务热情 / 环境舒服 / 出品不错 / 还会再来

3. 已帮你整理好
   可直接用，也可以改成自己的话
   [可编辑文案]

4. 复制并打开美团店铺
```

## Interaction Details

### Platform Selection

平台按钮显示消费来源，而不是平台名：

| `platformCode` | 当前平台名 | 新按钮文案 |
| --- | --- | --- |
| `meituan` | 美团 | 美团团购来的 |
| `dianping` | 大众点评 | 大众点评到店的 |
| `xiaohongshu` | 小红书 | 小红书看到来的 |
| `douyin` | 抖音 | 抖音团购来的 |

如果后台配置了自定义平台名，优先使用商家平台名，但按钮仍应表达“来源”，例如“从{platformName}来的”。

### Tag Selection

标签不是“关键词管理”的暴露，而是帮顾客省脑子的回答按钮。

建议默认最多展示 4-6 个，优先选最容易让顾客确认的体验点：

- 团购划算
- 服务热情
- 环境舒服
- 出品不错
- 还会再来
- 朋友推荐

标签标题：

```text
这次体验想提哪点？
```

不要写：

- “请选择关键词”
- “请选择标签”
- “个性化生成”

### Review Textarea

标题：

```text
已帮你整理好
```

辅助文案：

```text
可直接用，也可以改成自己的话。
```

textarea 保持可编辑，不要做成只读卡片。可编辑本身就是信任信号。

### Action Buttons

主按钮：

```text
复制并打开{平台名}店铺
```

次按钮：

```text
换个说法
```

单独“复制”降级为次级能力，或者在复制失败时作为备用路径出现。第一屏不要并列展示两个同等重量按钮。

### Completion State

点击主按钮后：

1. 先复制 textarea 内容。
2. 上报 `platform_link_click`，带 `editedContent`。
3. 打开平台店铺页。

如果复制成功但跳转失败，提示：

```text
文案已复制。店铺页暂时打不开，可以手动打开 App 后粘贴。
```

如果复制失败，提示：

```text
没有自动复制成功，请长按文案复制后再打开店铺。
```

不提示“找评价入口”的路径。

## State Table

| State | User Sees | Primary Action | Notes |
| --- | --- | --- | --- |
| Loading | 店名和步骤骨架 | None | 首屏不能白屏 |
| No platform links | 当前门店还没有可用平台 | 联系店员 | 不出现技术错误 |
| Platform selected, no review yet | 正在整理评价 | Disabled CTA | 平台选择后立刻拉取文案 |
| Review available | 已帮你整理好，可改成自己的话 | 复制并打开店铺 | 主成功路径 |
| No review stock | 暂时没有可用文案，请找店员处理 | 找店员 | 不让顾客以为自己操作错 |
| Copy failed | 请长按文案复制 | 打开店铺 | 保留跳转能力 |
| Link missing | 该平台暂时没有店铺链接 | 换一个来源 | 商家配置问题，不怪顾客 |
| Event logging failed | 用户无感 | Continue | 统计失败不能阻塞顾客 |

## Analytics And Event Contract

继续沿用当前动作：

- `page_view`
- `platform_select`
- `review_pick_by_tag`
- `review_switch`
- `review_copy`
- `platform_link_click`

需要调整产品解释：

- 商家后台仍只把 `platform_link_click` 视为“引导发布次数”。
- `review_copy` 是辅助指标，不作为商家主价值展示。
- `review_switch` 是负反馈，继续用于后续生成规避。
- `editedContent` 继续保存，用于学习真实顾客愿意留下的表达。

## Open Questions

- 不同平台的“来源按钮”是否允许商家自定义，比如“美团团购来的”改成“美团套餐来的”。
- 是否需要记录顾客是否先选了标签，再点击主按钮，用来判断标签是否真的降低动脑成本。
- 小红书和抖音场景是否需要更强图片素材入口；第一版先不放大这个问题。
- 如果商家只有一个平台，是否跳过来源选择，直接进入体验点选择。建议第一版仍保留平台区，但单平台默认选中。

## Success Criteria

- 顾客从打开页面到点击主按钮，目标小于 10 秒。
- 服务员不需要解释页面怎么用，只需要发起请求和协助平台内评价入口。
- 顾客能清楚知道文案可以改成自己的话。
- 商家后台“累计引导发布”能随着真实主按钮点击增长。
- 真实门店观察中，顾客卡住点不再是“选哪个平台”或“怎么写评价”，而只剩平台 App 内评价入口。

## Distribution Plan

这是现有 Web/H5 落地页改版，不新增分发渠道。

- NFC 卡继续指向 `/landing/:token` 或门店 uuid 落地地址。
- 服务器现有部署链路继续承载前端静态资源和 Go backend API。
- 后续实现应保持 `/ppk` 子路径部署兼容，不能破坏 `import.meta.env.BASE_URL` 和公开 API base path。

## Dependencies

- 商家后台必须已配置可用平台链接。
- 对应平台评价池必须有可发放文案，或 agent-service 可补池。
- 店员话术需要同步给门店执行，否则页面会失去前置语境。
- 文案生成质量必须足够自然，不能让顾客产生“这是假的”的第一反应。

## The Assignment

上线前不要先做大视觉。先拿这个 B 版流程做一次线下观察：

在一家真实门店，服务员只说两句固定话术，让 5 个真实顾客独立完成页面操作。记录每个人：

- 从打开页面到点击主按钮用了几秒。
- 有没有改文案。
- 有没有点“换个说法”。
- 卡在页面内，还是卡在平台 App 内。
- 有没有说“这不像我写的”或“这个可以直接发”。

如果 5 个人里至少 3 个能在 10 秒内点主按钮，才继续做视觉 polish。否则先修流程。

## What I Noticed About How You Think

- 你纠正了“平台由商家先选好”这个假设，说“顾客是愿意去写评论的，所以会自己选择”。这是对真实场景的判断，不是对后台结构的幻想。
- 你把最大阻力说得很准：“找 App 麻烦、动脑子写评论麻烦”。这让页面目标从“生成内容”变成“减少顾客操作成本”。
- 你拒绝在页面写回报，把回报留给服务员和顾客现场互动。这个边界很重要，它让页面保持真实评价辅助，而不是好评换利益。
