# 商家评论生成优化设计

## 背景

碰碰卡的核心价值不是一次性生成几条好评，而是让商家长期使用后，仍然能持续得到自然、不重复、可发布的评论文案。

上一版已经把商家后台的生成偏好接入 agent，包括重点方向、语气、参考评论和字数波动。但只靠几个语气选项，长期使用后仍然容易出现同质化：

- 评论身份相似：都像同一个人在写。
- 评论场景相似：都从“朋友推荐/第一次来”这类角度开始。
- 评论内容相似：总是在夸同一个菜品或同一种服务。
- 评论结构相似：开头、转折、结尾都像模板。
- 评论长度相似：一批评论看起来像同一个模型批量生成。

因此当前优化的目标是：商家只做轻量选择，系统自动把评论维度打散，让 20 条评论队列内部自然形成差异。

## 设计原则

1. 商家选大方向，不让商家配置细碎角色。
2. 小方向由系统分配，避免增加商家操作负担。
3. 每批评论需要在身份、场景、内容、结构、字数上分散。
4. 历史用户反馈要进入下一批生成，接受的评论用于学习，不喜欢的评论用于规避。
5. 所有偏好只影响写作方向，不突破平台规则、真实性约束和禁用词约束。

## 当前内容维度

| 类型 | 大方向 | 小方向/内容来源 | 是否商家选择 | 作用 |
| --- | --- | --- | --- | --- |
| 门店基础 | 店名 | 门店名称 | 后台填写 | 保证评论指向真实门店 |
| 门店基础 | 行业 | 餐饮、足疗、美甲、美发等 | 后台填写 | 控制行业词汇，避免串行业 |
| 门店基础 | 简介 | 店铺介绍、特色说明 | 后台填写 | 给 agent 理解门店卖点 |
| 门店基础 | 地址 | 门店地址 | 后台填写 | 允许评论出现合理位置感 |
| 门店基础 | 品牌语气 | 轻松自然、克制真实等 | 后台填写 | 给整体表达定调 |
| 重点内容 | 重点方向 | 菜品、服务、环境、上菜快、适合聚餐等 | 商家选择或新增 | 让评论自然提到商家想突出的点 |
| 评论风格 | 语气 | 自然随手写、细节丰富、年轻口语、稍微克制、像老顾客 | 商家选择 | 控制表达口吻 |
| 多样化方向 | 顾客身份 | 新客、老顾客、附近上班族、朋友聚餐同行者、情侣约会的一方、家庭聚餐成员、外地游客或顺路打卡 | 商家只选大方向 | 避免每条评论都像同一类顾客 |
| 多样化方向 | 到店场景 | 工作日午餐、下班后晚餐、周末小聚、临时路过、朋友约饭、家庭聚餐、排队后入座 | 商家只选大方向 | 让评论开头和场景更分散 |
| 多样化方向 | 内容角度 | 招牌菜记忆点、服务细节、环境感受、性价比、出餐速度、复购理由、小缺点再转满意 | 商家只选大方向 | 避免内容都在夸同一个点 |
| 多样化方向 | 表达结构 | 到店原因开头、第一口感受开头、同行人反应开头、预期对比开头、下次想点什么结尾、两段短评、一句短结论 | 商家只选大方向 | 避免句式和结构重复 |
| 参考样本 | 参考评论 | 商家贴真实顾客评论 | 商家填写 | agent 只学习节奏、细节密度、口语程度，不照抄 |
| 字数控制 | 字数波动 | 短评、中评、长评 | 系统自动 | 避免所有评论长度接近 |
| 用户反馈 | 喜欢样本 | 被复制或接受的评论 | 系统收集 | 后续生成学习用户喜欢的表达 |
| 用户反馈 | 不喜欢样本 | 被“换一换”的评论 | 系统收集 | 后续生成规避类似套路或空泛表达 |
| 库存控制 | 评论队列 | 目标 20 条，低于 10 条自动补充 | 系统自动 | 保证用户每次进入都有可用评论 |

## 商家后台交互

商家后台的“优化下一批评论”区域承担生成偏好配置：

- 重点方向：从商家关键词和推荐标签中选择，也可以手动新增。
- 多样化方向：选择顾客身份、到店场景、内容角度、表达结构。
- 语气：选择自然随手写、细节丰富、年轻口语、稍微克制、像老顾客。
- 参考评论：粘贴真实顾客评论，供 agent 学习表达节奏。

商家不需要选择“新客、老客、附近上班族”这些小项。商家只需要选择“顾客身份”这个大方向，系统会在生成每条评论时分配具体小方向。

## Agent 生成策略

### 1. 大方向进入生成契约

商家偏好通过 `generation_preferences` 传给 agent-service：

| 字段 | 含义 |
| --- | --- |
| `focus_keywords` | 商家希望评论自然提到的重点 |
| `style_codes` | 评论语气方向 |
| `diversity_dimensions` | 商家选择的多样化大方向 |
| `reference_reviews` | 商家提供的参考评论 |
| `length_variance` | 字数波动策略 |

### 2. 小方向由系统分配

当前实现不是纯随机，而是按评论序号稳定轮转小方向。

这样做有两个好处：

- 一批 20 条评论更容易均匀覆盖不同身份、场景、角度和结构。
- 测试和线上问题复现更稳定，不会因为随机数导致同样输入每次表现完全不同。

如果后续需要更强随机感，可以在稳定轮转基础上加入“门店 ID + 批次 ID”的种子随机，而不是完全无约束随机。

### 3. Prompt 中的约束

agent prompt 会加入“本条多样化视角”，例如：

```text
本条多样化视角：
- 顾客身份：附近上班族
- 内容角度：先写服务细节
```

这些内容是写作视角，不要求文案机械自我介绍。比如不应该写成“我是附近上班族，我来吃饭”，而应该自然表现为“中午从公司过来吃一顿，出餐挺快”。

## 反馈闭环

用户行为会反向影响下一批评论：

| 用户行为 | 系统含义 | 后续生成处理 |
| --- | --- | --- |
| 复制或接受评论 | 用户认可这条评论 | 作为 accepted 样本，学习真实细节和自然口吻 |
| 点击换一换 | 用户不喜欢上一条评论 | 作为 rejected 样本，避免类似套路、空泛、夸张或重复表达 |
| 平台发布点击 | 代表一次有效引导发布 | 进入商家后台数据曲线 |

反馈样本只用于总结写作方向，不允许 agent 照抄原句。

## 评论队列策略

评论队列的目标库存是 20 条。

- 当可用评论低于 10 条时，自动触发补充生成。
- 生成时带上商家偏好和历史反馈。
- 已复制或已发放的评论不应继续作为可发放库存重复出现。
- 每次补池都应该重新分配多样化小方向，避免长期只在同一批视角内循环。

## 后续可优化方向

1. 批次级生成计划：生成前先规划 20 条评论的身份、场景、角度、结构分布，再逐条生成。
2. 相似度过滤：入库前和最近已发布、已换掉、已接受评论做文本相似度检查。
3. 商家风格模仿：把店家自己写的评论单独作为风格参考，而不是混在普通参考评论里。
4. 平台差异化：小红书、点评、美团、抖音使用不同的结构分布和口语程度。
5. 数据看板闭环：用发布次数曲线观察优化是否提高商家持续使用频率。
